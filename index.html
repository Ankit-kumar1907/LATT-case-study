Latt Subscription Manager

Realigning system persistence with user intent through database logic and ethical UI.

RoleUX Research & Full Stack
Timeline4 Weeks (Solo Project)
Core TechNode.js, SQL, Vanilla JS

02. Research & Ethical Framework

The subscription economy thrives on "invisibility." Once a user subscribes, the system is designed to fade into the background. My goal was to invert this: How might we make financial commitments visible without being annoying?

The "Anxiety Gap" Finding

I conducted a comparative analysis of 15 subscription management tools. A distinct pattern emerged: trust drops significantly when applications request banking credentials (OAuth/Plaid) before demonstrating value.

Key Insight: Users treat payment processing not as a technical step, but as a moment of vulnerability. A system that hides logic during this phase increases anxiety.

03. Balancing Feasibility & Usability

The Automation Bottleneck: The ideal user experience for a subscription manager is "Zero-Input" discovery via open banking APIs (e.g., Plaid). However, the implementation costs and security compliance requirements (SOC2) rendered this unfeasible for the MVP architecture.

The Pivot: From Automation to Agency. Rather than shipping a broken or insecure automation feature, I pivoted to a Manual Entry Model. The design challenge shifted from "data display" to "data input efficiency."

By placing related fields (Currency + Amount) in a horizontal flex-row within the codebase, I reduced the vertical scanning distance, making the manual process feel less like data entry and more like a quick check-in.

04. The Engineering Logic

As a student of Mathematics, I approached the backend not just as data storage, but as a system of relations.

Data-Driven Interface Design

The interface is not just a layout; it is a direct visual manifestation of the underlying PostgreSQL schema. I utilized a Relational Database Management System (RDBMS) to normalize user data, separating Users, Services, and Subscriptions to ensure data integrity (3NF).

Query-Driven Layout: The "Upcoming Bills" card is the result of an SQL query (`ORDER BY renewal_date ASC`). This ensures the user's Mental Model (prioritizing immediate debts) aligns perfectly with the System Model (temporal sorting of data).

The "Game Loop" Analogy

In code, handling recurring payments is similar to handling input in a game loop. When building a racing game, we don't write code for every single time the player presses the horn. Instead, we write a Callback Function that listens for the event.

Conceptual Logic:
Just as the game listens for onPress(horn), the Latt backend listens for onDate(renewal).

Instead of hard-coding dates, the system uses a cron-job (the game loop) to check for renewal_date <= today and triggers the processPayment() callback.

06. Intentionality & Micro-interactions

Every pixel must have a purpose. I specifically focused on the "Cancel Subscription" flow to ensure the design did not rely on dark patterns, but rather on clear communication.

Rationale: I leveraged the user's existing mental model where red indicates a destructive action. However, a solid red button creates excessive visual weight. By utilizing an outline style (Ghost Button), I maintained the caution signal acting as a necessary friction point, forcing the user to pause, without dominating the interface.

07. Resilience & Failure Analysis

In HCI, the "Happy Path" (when everything works) is easy. The real challenge is the "Unhappy Path."

Constraint-Based Design

Critique of Generic Feedback: High cognitive load occurs when errors are vague ("Error!"). This forces the user to scan the entire form to locate the mistake.

The Solution (False Affordance): In early iterations, the "Continue" button remained active during errors. The final design disables the primary action until valid input is detected. Furthermore, addressing Network Errors, I added an 'Exit' button to satisfy Nielsen‚Äôs heuristic of User Control and Freedom.

08. Future Roadmap: Defining the MVP

A key part of engineering is knowing what not to build.

Desirability vs. Feasibility: For complex features like PDF parsing (OCR), I implemented "Wizard of Oz" placeholders. By using hard-coded alerts to signal future intent, I established a clear boundary between the MVP and the V2 roadmap, avoiding technical debt on unvalidated features.

10. Reflection & Learnings

Building Latt taught me that UX is dependent on Backend Architecture. You cannot design a "Transparent" UI if your database is opaque.

If I were to continue this project, I would implement the Plaid API to move beyond manual entry, but only after the user has manually tracked their first subscription, maintaining the trust-first onboarding flow.
